Two things fail silently in an AI-native engineering organisation: the regulatory evidence nobody has time to assemble, and the scaffolding the agents themselves run inside. This is a catalogue of skills that makes both of them visible, testable, and routine.

The problem

Both failures are invisible until they are expensive.

Failure mode 01
Compliance fails through friction.

Nobody regenerates a software bill of materials mid-sprint. Nobody triages a CVE the week a release ships. It is not ignorance — it is that the work is tedious and always less urgent than the feature. So the evidence is assembled in a panic, months late, by whoever is unlucky. The regulatory clock does not care: CRA vulnerability-reporting obligations begin 11 September 2026.

Failure mode 02
The harness fails through invisibility.

The instructions, skills and guardrails an agent works inside will rot, and nothing will tell you. A CLAUDE.md grows until the agent quietly stops obeying half of it. A skill's trigger drifts until it fires on everything, or on nothing. There is no error message and no alert. The agent simply gets worse — and the team concludes the model got worse.

What changes

Make the work produce its own evidence.

Not a policy, not a training course, not a quarterly audit. Seven skills that sit inside the repositories where the work already happens, and do the tedious half automatically — leaving humans the one job they are actually needed for: deciding.

Efficiency

Cost becomes a decision, not an accident

Teams hit their usage limits before the day is done, and blame the model. The real cause is almost always a session nobody cleaned. Routing work to the right model tier — and delegating bulk reading to cheaper, isolated agents — becomes a written repository rule rather than a matter of individual discipline.

Structure

The scaffolding becomes an artifact

The agent setup in each repository stops being folklore that one person configured once and nobody has touched since. It gets inventoried across eight surfaces and reported on, with every gap named and assigned to whoever owns the fix.

Quality

Claims are tested, not asserted

A skill nobody tested probably does not work — or worse, fires constantly and looks like it does. Every trigger in this catalogue is put in front of independent blind reviewers, and a disagreement between them is recorded as a defect rather than rounded away.

Predictability

Junior work looks like senior work

The discipline a staff engineer applies by instinct — plan first, delegate the right thing, verify before claiming done — stops depending on who happens to be at the keyboard. It is encoded once, and applies to everyone.

Compliance

Evidence, on demand, with a name on it

A bill of materials, a supply-chain diff, a vulnerability triage and a regulatory gap report, generated in one invocation and handed to a human for approval. Draft by machine, approved by person — with the approval recorded.

Governance

Nothing marks its own homework

No artifact in this system is ever labelled Accepted, Compliant or Validated by the machine that produced it. Everything arrives as a draft. A person decides. That is the whole contract, and it is not negotiable.

Reported states are always present gap not applicable — each with a stated reason and a pointer to the evidence. A bare “looks good” is treated as a bug, not a summary.

The catalogue

Seven skills, two tracks.

A skill is a folder Claude Code loads on demand: a procedure, plus the scripts and templates it needs. It self-triggers from its own description — which means the description is the router, and is the only thing seen at trigger time.

SkillTrackAnswers
harness-auditharness What scaffolding does this repository give an agent, and what is missing? Inventories eight surfaces. Entry point.
claude-md-authoringharness Is this CLAUDE.md policy, or a skill in disguise? Diagnoses “the agent keeps ignoring X” — where the answer is almost never a stronger instruction.
subagent-authoringharness Should this be a skill, a subagent, a slash command, or a hook? The most common mistake is building a subagent where a hook was required.
harness-evalharness Does this skill actually fire? Did that change actually help?
model-routingharness Which model tier for which work — and why the usage limit runs out before the day does.
cra-evidencecompliance Bill of materials, release-to-release supply-chain diff, vulnerability triage, CRA Annex I gap report, accessibility scan — assembled into one reviewable dossier.
adr-managementcompliance Why is the system like this? Drafts the decision record at the moment the decision is made, instead of never.

Skills install flat, to .claude/skills/<name>/. Claude Code knows nothing about tracks — the track directories organise the source repository only, so nothing already installed anywhere breaks.

The differentiator

A skill you did not test probably does not fire.

Or it fires on everything — which is worse, because it looks like it is working. Both failures are invisible from the inside. So every skill's trigger is put in front of a quorum.

Judge 01 · blind
Sees only the skill's description and one prompt. Never the skill body — because the body is invisible to the real router too.
Judge 02 · blind
Dispatched with no tools. A judge that can read the repository can look up the answer, and defeats the whole test silently.
Judge 03 · blind
One judge, one prompt, one vote. A judge shown the whole set calibrates — “about half of these must be triggers” — and inflates its own score.
A split vote is flaky — never rounded up to a pass.

This is the rule the entire method rests on. Triggering is a probabilistic decision: a single judge returns one sample of it and calls that evidence. Three judges can disagree — and the disagreement is the finding. It is precisely what an ambiguous trigger looks like from the outside, and a lone reviewer structurally cannot see it. The moment a 2-of-3 gets rounded into a pass to make a table look clean, the exercise becomes theatre with a spreadsheet attached.

372 blind judgements to date; final state 52 of 52 rows unanimous. The ledger keeps the rounds that failed — the split votes, the dissenting reasoning, and a “clean sweep” claim that was later found false and retracted in place. The clean final number is not the interesting part of that file. The rounds before it are.

Verification

Tested against a repository built to be broken.

An audit tool validated only against a healthy repository proves nothing. So the fixture is deliberately defective — a bloated instruction file, no verify gate, no hooks, and a skill whose description triggers on any prompt. The tooling has to catch all of it. Its README says, in as many words: do not fix these defects — they are the test.

Install

One command, per repository.

git clone https://github.com/Oltrematica/oltrematica-skills.git /tmp/os
/tmp/os/scripts/install.sh harness-audit --to /path/to/your-repo

# then, in that repo, simply say:
"audit our harness"

The harness skills assume the Superpowers plugin is installed, and deliberately do not re-implement it. Planning, test-driven development, debugging and code review are its territory. This catalogue adds what it does not cover: what the harness is made of, and whether it works.
