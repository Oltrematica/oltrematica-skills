# CI gate proposal (Phase 3 — design only)

**Status: proposal. Nothing described here is wired into any repo.** This is
the Phase 3 deliverable from the program brief (§8); Phase 1–2 explicitly
excluded CI enforcement — the skill orchestrates and interprets evidence, a
human triggers it. This document sketches what an automated, unattended gate
would look like so it can be reviewed and adapted per repo when the team is
ready.

## Purpose

On every tagged release, CI should produce the SBOM and vulnerability scan
automatically, without a human running the skill first. The gate then fails
the release build on any unremediated **Critical** finding, so a critical
vulnerability can't ship silently even if nobody ran `cra-evidence` W2 by
hand.

## Proposed workflow

Adapt-ready GitHub Actions YAML. Not enforced anywhere yet — copy it into a
repo's `.github/workflows/` deliberately, don't assume it exists.

```yaml
# .github/workflows/release-evidence.yml  (PROPOSAL — not enforced anywhere yet)
name: release-evidence
on:
  push:
    tags: ['v*']
permissions:
  contents: write
jobs:
  sbom-and-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Generate SBOM (CycloneDX)
        uses: anchore/sbom-action@v0
        with:
          format: cyclonedx-json
          output-file: sbom-${{ github.ref_name }}.cdx.json
          artifact-name: sbom-${{ github.ref_name }}.cdx.json
      - name: Scan SBOM
        uses: anchore/scan-action@v6
        with:
          sbom: sbom-${{ github.ref_name }}.cdx.json
          fail-build: true
          severity-cutoff: critical
      - name: Attach SBOM to release
        uses: softprops/action-gh-release@v2
        with:
          files: sbom-${{ github.ref_name }}.cdx.json
```

## Gate policy discussion

Starting policy: `severity-cutoff: critical`. Anything below Critical does
not fail the build — it still needs triage, but through the normal `W2`
review flow, not as a release blocker.

Escape hatch, needed before this can be turned on for real: a triage record
with an `accept` outcome (see `triage_guidance.md`) plus an ADR documenting
the risk-acceptance decision should be able to unblock a release without
raising the cutoff globally. The mechanism for that — an allowlist file the
workflow reads before failing the build — is a design note here, not
something built. It would need at minimum: CVE ID, expiry or re-review date,
link to the approving ADR.

## Relationship to the skill

CI, if adopted, produces raw evidence (SBOM + scan output) unattended on
every tag. The `cra-evidence` skill remains the interpretation and triage
layer — it's what turns a raw grype finding into a human-reviewable
`Proposed` triage record, updates the dossier, and hands architecturally
significant decisions to `adr-management`. The two are not redundant: CI
guarantees evidence *exists* even if nobody remembered to ask for it; the
skill is what makes that evidence legible and actionable. They share
formats deliberately — CycloneDX for the SBOM, grype's JSON output for
findings — so evidence generated by CI and evidence generated by the skill
are interchangeable inputs to the same triage process.

## Open points for Mircha/Alex review

- Pin action versions (`@v4`, `@v0`, `@v6`) exactly, or track major tags as
  drafted above? Pinning is safer for supply-chain integrity, tracking major
  tags means fewer maintenance PRs.
- Where does the allowlist file live, and who can edit it (branch
  protection implications)?
- Should the SBOM produced by CI also be committed back into the target
  repo's `compliance/sbom/` directory, so it lands in the same place as
  SBOMs generated manually via the skill? If not, the dossier needs a
  pointer to the CI artifact instead.
